Mental Models

Partitioning usually means splitting data inside one logical database system. Sharding means splitting data across multiple database nodes that are queried as one logical dataset.
Why partition
  • Reduce index and table working set.
  • Improve parallelism and prune scans.
  • Support lifecycle operations (archive/drop old partitions).
Why shard
  • Break single-node CPU, IOPS, and storage limits.
  • Isolate noisy tenants/workloads.
  • Enable horizontal scaling and failure blast-radius control.
Golden rule
  • Pick a distribution key aligned to your hottest read/write paths.
  • Minimize cross-partition and cross-shard joins/transactions.
  • Design for rebalancing from day one.

Partitioning Strategies (mostly relational)

Strategy Best For Main Risk
Range Time-series/event data, retention windows Hot newest partition
Hash Even distribution for high cardinality keys Poor locality for range scans
List Bounded categorical splits (region, tier) Skew if category distribution shifts
Composite Mixed filters (time + tenant) Operational complexity

In PostgreSQL/MySQL workloads, partitioning helps first before sharding. It is often cheaper to tune schema, indexes, and partition pruning than to introduce distributed query complexity.

Watch out: many teams partition tables but keep globally unique secondary indexes or cross-partition queries everywhere, which cancels expected gains.

NoSQL note: DynamoDB, Cassandra, and similar systems expose partition keys directly; your schema design is effectively sharding-by-design. Bad key choice causes hot partitions immediately.

Sharding Strategies

Hash-based sharding

  • Route with shard = hash(key) % N.
  • Great write spread and simple routing.
  • Resharding is expensive without virtual shards.

Range-based sharding

  • Good for key locality and bounded scans.
  • Needs split/merge logic for hot ranges.
  • Requires active hotspot detection.

Directory / lookup sharding

  • Mapping service decides shard for each tenant/key.
  • Most flexible for online migration.
  • Adds control plane and consistency concerns.

Recommended production pattern: use many virtual shards (logical buckets) mapped to fewer physical nodes. Rebalancing then moves bucket ownership rather than rewriting hash functions globally.

Schema tip: include tenant or shard key in every major table and index prefix to keep reads/writes shard-local.

Operations: Rebalancing, Migrations, and Failure Modes

  • Online migration pattern: dual-write + backfill + read-verify + cutover + decommission.
  • Reshard trigger metrics: p99 latency by shard, disk growth slope, queue lag, compaction debt, and replica catch-up lag.
  • Cross-shard transactions: avoid if possible; use sagas, idempotent writes, and compensating actions.
  • Observability minimum: shard-level SLO dashboard, top keys by QPS, hot partition alarms, rebalance progress metrics.
Common failure mode: teams shard only data storage, but leave cache keys, job queues, and downstream services unsharded. The real bottleneck just moves.

System Design Interview Cheat Sheet

1) Clarify first

  • Read/write ratio?
  • Access pattern by key/time/tenant?
  • Strong vs eventual consistency requirements?

2) Capacity math

  • Estimate QPS peak and growth.
  • Estimate per-row size and index overhead.
  • Compute hot-key risk and per-node headroom.

3) Key decisions

  • Partition key and shard key.
  • Routing layer and metadata store.
  • Rebalance strategy and cutover protocol.

4) Trade-offs to say out loud

  • Locality vs even distribution.
  • Operational simplicity vs flexibility.
  • Consistency guarantees vs write throughput.
Interview line that scores well: "I'll optimize for shard-local reads/writes first, then explicitly design rebalancing and hot-key mitigation because scale failures are usually operational, not theoretical."

War Stories (Anonymized)

"Hash looked perfect - until one customer wasn't"

A B2B SaaS team hashed by tenant_id. One enterprise tenant generated 35% of writes and saturated a single shard.

Lesson: model "tenant size skew," not just tenant count. Add sub-sharding for whale tenants.

"Range shards melted every Monday morning"

Time-range sharding pushed all new traffic to latest shard. Weekly batch jobs amplified hotspotting.

Lesson: combine time with hash bucketing (e.g., day + bucket) for parallel ingest.

"Rebalance succeeded, latency still doubled"

Data moved successfully, but cache and async workers still targeted old shard mapping for hours.

Lesson: shard migration runbooks must include cache invalidation, worker rollout, and mapping TTL controls.
