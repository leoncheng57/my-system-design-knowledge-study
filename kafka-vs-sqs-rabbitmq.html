1. Fundamental Mental Models

The one sentence that matters: Kafka is a distributed append-only log where consumers control their read position. SQS/RabbitMQ are queues where the broker controls message lifecycle. This single distinction cascades into every architectural decision.
Log-based broker

Kafka / Kinesis / Pulsar

Messages are written sequentially to disk (append-only segments). Consumers maintain an offset — a pointer into the log. The broker never deletes messages on read; retention is time-based or size-based. This means:

  • Replay: Seek to any offset and re-consume
  • Multiple consumer groups: Each tracks its own offset independently
  • Ordering: Guaranteed per-partition (sequential disk reads)
  • Throughput: Sequential I/O + zero-copy (sendfile) = millions of msg/s
In-memory message broker

RabbitMQ / SQS / Redis Streams*

Messages live in an in-memory data structure (linked list, heap, etc.) and are deleted after acknowledgment. The broker pushes (RabbitMQ) or the consumer polls (SQS). This means:

  • No replay: Once acked, it's gone
  • Competing consumers: Each message goes to exactly one consumer
  • Routing: RabbitMQ has rich exchange-based routing (direct, topic, fanout, headers)
  • Simplicity: Great for task/job queues — fire and forget

* Redis Streams is a hybrid — it has log-like consumer groups but is primarily in-memory.

2. Kafka Architecture & Internals

Log-Based Message Broker — How It Actually Works

Fig 1. Kafka's log-based architecture. Consumers independently track offsets — Consumer B is 60 seconds behind Consumer A.

Why Sequential Disk I/O Matters

Counter-intuitively, Kafka uses disk, not RAM, as its primary storage — and it's faster than in-memory brokers at scale. Here's why:

  • Sequential writes to an append-only log hit ~600 MB/s on modern SSDs (vs ~100 KB/s for random writes)
  • OS page cache — the kernel automatically caches recently written data in RAM. Consumers reading "near the tip" read from cache, not disk
  • Zero-copy via sendfile() — data goes from page cache directly to the network socket, bypassing userspace entirely
  • Batching — producers batch writes, reducing syscall overhead. Compression (snappy, lz4, zstd) happens at the batch level

Partition Replication

Fig 2. Kafka replication. The leader handles all I/O; followers pull asynchronously. ISR set determines durability guarantees.
Consumer side — reading is not taking: In Kafka, consumers do not remove messages from the partition. A consumer polls the partition leader for records starting at its current offset, processes the returned batch, then commits a newer offset to say "I've processed up to here." The log remains unchanged until retention deletes old segments, which is why another consumer group can read the same messages independently or an existing group can rewind and replay. This differs from SQS/RabbitMQ, where ACKing a message removes it from the queue.
Senior SWE pitfall — acks configuration:
acks=0 — fire-and-forget (fastest, data loss possible).
acks=1 — leader writes to its local log (leader failure = data loss if not yet replicated).
acks=all — waits for all ISR replicas. Combined with min.insync.replicas=2, this is the standard durability config for production. If ISR shrinks below min.insync.replicas, the partition becomes read-only.

3. Traditional Queue Internals (SQS / RabbitMQ)

Fig 3. Traditional queue: in-memory linked list. Messages are deleted on ACK. Multiple consumers = competing consumers (each msg to one consumer only).

SQS Visibility Timeout — The Hidden Complexity

When a consumer receives an SQS message, it becomes invisible to other consumers for a configurable duration (VisibilityTimeout, default 30s). If the consumer crashes without ACKing, the message reappears after timeout — this means at-least-once delivery is the default. Your consumer must be idempotent.

4. Delivery Semantics & Failure Handling

Semantic Definition Who supports it Production implication
At-most-once Message may be lost, never redelivered Kafka (acks=0), RabbitMQ (no-ack mode) Acceptable for metrics/logging where some loss is OK
At-least-once Message delivered 1+ times, never lost All (default for Kafka acks=1/all, SQS, RabbitMQ) You must make consumers idempotent. Use dedup keys, upserts, or idempotency tokens
Exactly-once Message processed exactly once, no duplicates Kafka (idempotent producer + transactions), SQS FIFO (dedup ID) Kafka EOS requires enable.idempotence=true + transactional API. Significant perf overhead. SQS FIFO dedup window = 5 min
The senior-engineer take on "exactly-once": True exactly-once delivery across distributed systems is impossible (see: Two Generals Problem). What Kafka offers is effectively exactly-once within its transactional boundary — idempotent produces + atomic writes across partitions. The moment you involve an external system (database, API call), you're back to at-least-once and need application-level idempotency.

Dead Letter Queues — What Happens When Processing Fails

SQS

First-class DLQ support. Configure maxReceiveCount — after N failed receives, message auto-moves to a designated DLQ. You can redrive messages back from the DLQ via the AWS console.

RabbitMQ

Built-in via x-dead-letter-exchange queue argument. Messages moved on rejection (basic.nack), expiry (TTL), or queue length overflow. Supports re-routing to any exchange.

Kafka

No built-in DLQ. You implement it yourself: catch processing errors in your consumer, produce the failed message to a .dlq topic, commit the offset, and move on. This is application-level code you own.

6. Backpressure & Flow Control

Kafka

Consumer-controlled pull. Consumers fetch batches at their own rate. If a consumer is slow, it just falls behind (consumer lag increases). The log doesn't care — messages sit on disk until retention expires.

  • Producer: buffer.memory + max.block.ms — producer blocks if internal buffer is full
  • Consumer: max.poll.records + max.poll.interval.ms — consumer is kicked from group if it doesn't poll fast enough
  • Monitoring: Consumer lag (offset difference) is your primary health metric

RabbitMQ

Broker-controlled push with prefetch. The prefetch_count limits unacked messages per consumer. If all consumers are at their prefetch limit, the broker stops sending — backpressure propagates to the producer.

  • Memory/disk alarms halt all publishers when resources are low
  • Flow control signals slow producers when queues grow too fast
  • Danger: Millions of messages in a queue = serious performance degradation (page to disk)

SQS

Infinite queue, no backpressure. SQS doesn't push back on producers — messages pile up indefinitely. Backpressure is entirely your problem (monitor ApproximateNumberOfMessagesVisible and scale consumers).

  • Standard queue: effectively unlimited throughput
  • FIFO: 300 msg/s per group, 3000 msg/s per queue (with batching)
  • Auto-scales internally — you never provision capacity

7. AWS Kinesis & Other Popular Tools

AWS Kinesis Data Streams

Think of Kinesis as "managed Kafka on AWS" — log-based, shard-partitioned, replayable. But with a very different operational model and pricing.
Fig 6. Kinesis Data Streams. Each shard has fixed read/write throughput — scaling means adding/splitting shards.

Kinesis vs Kafka — When to Choose Which

Choose Kinesis when...

  • You're all-in on AWS and want zero ops
  • You need tight Lambda integration (event source mapping)
  • Your throughput is moderate (< hundreds of shards)
  • You want Firehose for automatic S3/Redshift delivery
  • You don't need Kafka's ecosystem (Connect, Streams, ksqlDB)

Choose Kafka (or MSK) when...

  • You need higher throughput per partition (no fixed 1 MB/s limit)
  • You need longer retention (Kinesis max 365d, Kafka can do infinite with tiered storage)
  • You want Kafka Streams / ksqlDB for stream processing
  • You need Kafka Connect for CDC and integrations
  • You want multi-cloud or on-prem flexibility

Other Popular Tools at a Glance

Log-based

Apache Pulsar

Kafka alternative with separated storage (BookKeeper) from serving (brokers). Supports multi-tenancy, geo-replication, and both queue + log semantics natively. Key differentiator: you can have millions of topics without partition overhead.

+ Strengths
  • Built-in multi-tenancy
  • Geo-replication
  • Unified queue + stream model
  • Better topic scalability
- Weaknesses
  • Smaller ecosystem than Kafka
  • More complex architecture (ZK + BK + brokers)
  • Less mature tooling
Message broker

NATS / NATS JetStream

Extremely lightweight, high-performance messaging. Core NATS is fire-and-forget pub/sub. JetStream adds persistence, replay, and exactly-once. Written in Go, single binary, sub-millisecond latency.

+ Strengths
  • Ultra-low latency (<1 ms)
  • Simple ops (single binary)
  • Built-in clustering & super-cluster
  • Great for edge/IoT
- Weaknesses
  • JetStream still maturing
  • Smaller community
  • No rich routing like RabbitMQ
In-memory + persistence

Redis Streams

Hybrid: Append-only log with consumer groups, living in Redis. Great when you already have Redis and need lightweight streaming. Consumer groups support ACK, pending lists, and claiming (for recovery).

+ Strengths
  • Already in your stack (Redis)
  • Consumer groups + replay
  • Very fast (in-memory)
  • Simple API (XADD, XREADGROUP)
- Weaknesses
  • Limited by memory (no disk tiering)
  • No partitioning across nodes
  • Not designed for massive throughput
  • Durability depends on Redis persistence config
Managed

Google Cloud Pub/Sub

GCP's fully managed messaging service. Semantically between SQS and Kafka — push or pull, at-least-once, automatic scaling. Subscriptions can replay from a snapshot/timestamp. Supports ordering keys.

+ Strengths
  • Fully managed, global
  • Push + pull delivery
  • Message replay (snapshots)
  • Ordering keys
- Weaknesses
  • GCP lock-in
  • Higher latency than Kafka
  • Less ecosystem than Kafka
  • Max 10 MB message

8. Use Cases: Log-Based vs Memory-Based Brokers

Log-based broker

Best use cases for Kafka / Kinesis / Pulsar

  • Event sourcing: Store immutable facts such as orders, payments, or inventory changes for later replay.
  • Audit trails: Keep a durable, ordered history of security events, financial events, or user activity.
  • Change Data Capture: Stream database changes with tools like Debezium and fan them out to search, cache, and analytics systems.
  • Real-time analytics: Feed metrics, clicks, logs, and telemetry into stream processing jobs or data lakes.
  • Multiple independent subscribers: Let fraud, notifications, BI, and ML consumers read the same events at their own pace.
  • Backfills and reprocessing: Re-run consumers from older offsets after bug fixes, schema migrations, or new feature launches.
Memory-based broker

Best use cases for RabbitMQ / SQS / Redis Streams

  • Background jobs: Queue work such as email sends, image resizing, report generation, and webhook delivery.
  • Work distribution: Spread independent tasks across a pool of competing workers where each task should run once.
  • Request/reply workflows: Implement RPC-style interactions with correlation IDs, reply queues, and timeouts.
  • Content-based routing: Use RabbitMQ exchanges to route messages by topic, header, region, tenant, or priority.
  • Retry and DLQ handling: Move failed work through retry queues, visibility timeouts, and dead letter queues.
  • Short-lived operational buffers: Absorb brief traffic spikes between producers and consumers without needing long-term replay.
Rule of thumb: choose log-based brokers when the event history itself is valuable; choose memory-based queues when the message is mainly a command to do work once.

10. Production War Stories & Gotchas

Kafka: Partition Rebalancing Storm

When a consumer joins/leaves a group, all partitions are reassigned (eager rebalancing). During rebalancing, no consumer in the group processes anything. With 500 partitions and a consumer restart, you can get 30+ seconds of downtime.

Fix: Use CooperativeStickyAssignor (Kafka 2.4+) for incremental rebalancing. Only revoked partitions stop processing.

Kafka: The "Poison Pill" Message

A malformed message crashes your consumer. Consumer restarts, re-reads the same message from the last committed offset, crashes again. Infinite loop.

Fix: Wrap processing in try/catch. On repeated failure, produce to a DLQ topic, commit the offset, and move on. This is not automatic in Kafka — you build it.

SQS: Invisible but Not Gone

Consumer receives message, starts processing, takes too long (exceeds VisibilityTimeout). Message becomes visible again. Another consumer picks it up. Now two consumers process the same message.

Fix: Make consumers idempotent. Or extend visibility timeout mid-processing via ChangeMessageVisibility.

RabbitMQ: Memory Alarm Death Spiral

Queue depth grows, RabbitMQ starts paging messages to disk, performance tanks, consumers slow down, queue grows faster. Eventually the memory alarm fires and all publishers are blocked.

Fix: Set queue length limits (x-max-length), use lazy queues for large backlogs, and monitor queue depth aggressively. Don't let RabbitMQ queues grow unbounded.

Kinesis: The "Hot Shard" Problem

If your partition key has low cardinality (e.g., "region" with 3 values), most data lands on 1 shard. That shard hits the 1 MB/s write limit while others idle.

Fix: Use a high-cardinality partition key (user_id, request_id) or add random suffixes. Note: this is the exact same problem as Kafka's "hot partition."

All: The Idempotency Gap

At-least-once delivery is the default everywhere. If your consumer INSERTs a row and crashes before ACKing, the message is redelivered and you get a duplicate row.

Fix: Use UPSERT / INSERT ... ON CONFLICT, or store a processed message ID set, or design for idempotent operations (set operations, not increments).

11. System Design Interview Cheat Sheet

The winning pattern: Don't just name-drop a technology. State the requirement, map it to the property of the system, then pick.

Example: "We need multiple downstream services to process the same order events independently, and if we add a new service next quarter, it should be able to backfill from the last 7 days. That requires message replay and independent consumer groups → Kafka (or Kinesis if we want managed on AWS)."
Requirement Best fit Why
Simple background job queue SQS Zero ops, built-in DLQ, pay-per-use, auto-scales
Event sourcing / audit log Kafka Immutable log, replayable, long retention
Multiple consumers for same events Kafka / Kinesis Consumer groups read independently
Content-based routing RabbitMQ Exchange bindings, topic patterns, headers
Request/reply RPC RabbitMQ Built-in reply queues, correlation IDs
Real-time analytics pipeline Kafka Kafka Streams / ksqlDB, high throughput, exactly-once
CDC (Change Data Capture) Kafka Kafka Connect + Debezium, log-based capture
Serverless event processing on AWS Kinesis / SQS Native Lambda event source mapping
Lightweight streaming (already have Redis) Redis Streams No new infrastructure, consumer groups, fast
Ultra-low-latency pub/sub NATS Sub-ms latency, single binary, simple
